# 全网写的最好的「缓存数据库一致性问题」

## 1. 先理解业务：一致性要求其实差别很大

不同的业务场景对一致性的要求差别其实很大。

### 1.1 低一致性场景

很多互联网业务其实并不要求高一致性，短时间的不一致不会对业务造成影响。例如**抖音热门视频点赞数、阅读量**

这些数据通常具备几个特点：

- 高并发读
- 高并发写
- 数据变化极快

但与此同时没有人会真正关心到底是 10w 还是 10w+1。因此系统通常会优先保证性能，而不是强一致性，在这种场景会采用「最终一致性」方案。

### 1.2 高一致性场景

但并不是所有业务都能接受不一致。

例如银行余额，你账户明明刚存了 1 万元，消费的时候由于读的是缓存的旧数据系统提示余额不足，诸如此类的问题会严重影响用户的信任，因此资金类业务通常会优先保证一致性，而不是极致性能。

## 2. 缓存一致性问题，到底是怎么产生的？

理想的情况下，只有单线程在执行，也就是说没有任何并发，且每个操作都保证成功，那保证缓存一致性是非常容易的，例如先更新DB后更新缓存都是可以的。

现实场景是，因为这是「A + B」两个步骤，所以有两种可能出现异常的场景：

1. **只有第一步成功，第二步失败**。（两个都成功最好；两个都失败也没关系，没有出现缓存数据库不一致的情况；第一步失败，自然代码层面就报异常了，不会去执行第二步，所以也不会产生缓存数据库不一致的情况；因此只有第一步成功，第二步失败的场景可能出现不一致。）
2. **并发**。由于网络存在延迟、抖动、丢包等问题，线程调度也不可控，所以在并发场景下可能会出现并发安全问题。

  场景1其实是比较好解决的，失败大不了就重试嘛，问题只是在于说重试的方式，怎么去做重试。

  场景2就不太好解决了，这类问题一出现，比较难排查，也不太好解决，所以我们尽量避免出现并发问题。

  我们需要基于现实情况，去分析这每一种方案的特点，才能知道每一种方案适合什么样的场景。

## 3. 为什么「更新缓存」方案容易出问题？

其实在设计缓存一致性方案的时候，更新缓存是一种最直觉的方案。既然数据库改了，那我把缓存也一起改掉不就行了？

甚至人们会觉得「更新缓存」比「删除缓存」更好。因为：

- 缓存里永远都是最新值
- 用户下一次读取时不需要重新查数据库
- 不会出现缓存为空

但实际上更新缓存远比删除缓存更危险，因为更新操作天然依赖**顺序**，而在高并发环境下，顺序恰恰是最难控制的东西。



我们从下面两个方案，展开讨论一下为什么「更新缓存」更容易出问题？

* 先更新缓存再更新数据库
* 先更新数据库再更新缓存

### 3.1 先更新缓存，再更新数据库

并发异常情况如图：

![并发异常-先改缓存后改DB](database-cache-consistency.assets/并发异常-先改缓存后改DB.svg)



出现这个并发异常的条件是，在线程B「执行完更新缓存之后」以及「更新完DB之前」线程A要执行完「更新缓存」+「更新DB」

其实这个异常出现的概率是非常高的，因为更新缓存是一个非常快的操作，线程A和B不会拉开多大的差距。所以可以理解为线程A和线程B以相同的起点，出发去抢DB的行锁，谁先抢到锁谁就先更新DB。

也就是说，无法保证最后一个更新DB的线程就是最后一个更新缓存的线程。

所以这个方案在并发场景下，出现「缓存数据库不一致」的概率是也比较高的。

### 3.2 先更新数据库，再更新缓存

并发异常情况如图：



![并发异常-先改DB后改缓存](database-cache-consistency.assets/并发异常-先改DB后改缓存.svg)

出现该异常需要满足的条件是在线程A「执行完更新DB之后」以及「更新完缓存之前」线程B要执行完「更新DB」+「更新缓存」

分析下来，出现异常的概率并不高。因为线程A写缓存是非常快的，在线程A更新完DB准备去写缓存到写完缓存这短短的几ms内，线程B要完成「更新DB」+「更新缓存」两个动作，可能性是比较低的。

为验证出现异常的概率，我跑了一个测试代码，反映出【**2**个线程并发写+先改DB后改缓存】的方案出异常的概率在**百分之20多**。

那是不是就是说就可以用这个方案呢？

别急。我们上面讨论的是非常理想的，只有两个写线程的场景。实际的业务场景肯定是不止两个线程，可能有十几个线程在并发的执行。为了验证线程数量比较多的场景下，该方案会出现异常的概率是多少，我也跑了一个测试代码，反映出【**20**个线程并发写+先改DB后改缓存】的方案出异常的概率在**百分之70多**。

只是增加了写线程的数量，为什么测试结果差距这么大？！

原因是：在测试代码中有20个写线程在并发执行写操作，需要保证最后一个写Redis的线程，必须是最后一个写DB的线程，才能够保证一致性。但是在高并发场景下，写Redis本身是一个非常快的操作，加上存在网络抖动、线程调度等因素，很难做到最后一个写Redis的线程，一定是最后一个写DB的线程。只要其它的线程因为网络抖动等各种原因稍微慢了一点，它们就可能是最后一个写Redis的线程。

也就是说最后写缓存的线程，未必是最后写数据库的线程。

以一种并不精确的方式计算，在2个线程的场景，出现不一致的概率是1/2。而在20个线程的场景，出现不一致的概率就是19/20。所以才会出现测试结果差距如此大。



我们可以看到不管是「先更新DB后更新缓存」还是「先更新缓存后更新DB」的方案，都非常依赖顺序。最后更新完DB的线程和最后更新完Redis的线程必须是同一个线程才能保证一致性。

但是在并发、CPU调度、网络IO等因素的共同作用下，完美的顺序是很难达到的，因此更新缓存的方案是不被推荐的。

## 4. 为什么业界更倾向「删除缓存」？

### 4.1 为什么删除更安全？

我们知道了更新缓存不被推荐，是因为更新缓存对顺序性有很高的要求。那删除缓存会更好吗？

删除操作有一个非常重要的特性：天然幂等。

什么叫幂等？简单理解就是执行一次和执行十次结果一样。

例如：

```
deleteCache()
deleteCache()
deleteCache()
```

最终结果仍然是缓存不存在。不会因为多删一次或者执行顺序不同导致数据错误。

因此删除缓存，比更新缓存更适合高并发场景。



我们根据「删除缓存」这种方案，分别对下面两种方案展开讨论

* 先删除缓存再更新数据库
* 先更新数据库再删除缓存

### 4.2 先删除缓存，再更新数据库

并发异常情况如图：

![并发异常-先删缓存后改DB](database-cache-consistency.assets/并发异常-先删缓存后改DB.svg)

出现该异常需要满足的条件是在线程A「执行完删除缓存之后」以及「更新完DB之前」线程B要执行完「读缓存」+「读DB」

这个情况出现的概率也很高，因为读DB比写DB快多了，因此在线程A写完DB前线程B「读完缓存」+「读完DB」可能性是很大的。

所以这个方案在并发场景下，出现「缓存数据库不一致」的概率是比较高的。

这个并发异常，是有解决策略的，就是**「延迟双删」**策略，伪代码如下：

```sql
#删除缓存
redis.delKey(X)
#更新数据库
db.update(X)
#睡眠
Thread.sleep(N)
#再删除缓存
redis.delKey(X)
```

出现异常的原因就是线程A删除了缓存之后，线程B读到旧数据并把旧数据又写回了缓存，线程A等一会再删一次缓存就好了。

既然要等就涉及到要等多久？

至少是要大于线程B「读完DB+写缓存」的耗时，但这个值在分布式和高并发场景难以准确评估，因为：

- DB 更新耗时不固定
- 网络延迟不固定
- 系统负载不固定
- GC 时间不固定

所以一般都是凭借经验大致估算这个延迟时间，例如延迟 1-5s，**只能尽可能地降低不一致的概率**，而无法做到保证强一致性。



### 5.4 先更新数据库，再删除缓存

并发异常情况如图：

![并发异常-先改DB后删缓存](database-cache-consistency.assets/并发异常-先改DB后删缓存.svg)

出现该异常需要满足的条件有：

* 读写并发
* 读缓存的时候缓存已失效
* 在线程A「读完DB之后」以及「写缓存之前」线程B要执行完「写DB」+「删除缓存」

这个情况出现可能性是非常低的，我们先只看条件3，我们知道写缓存本来就是很快的，写DB是很慢的。要在线程A写完缓存之前，线程B要完成「写DB+删除缓存」两个操作，可能性非常小。光光是条件3已经很难满足了，更加上条件2，要恰好线程A读缓存的时候缓存已失效，这就更难满足了。

所以这个方案在并发场景下，也能比较好得保证「缓存数据库一致性」。这个方案也是最被广泛推崇的方案，名为Cache Aside策略，旁路缓存策略。



这个方案也存在它的问题：

* 问题一：在高并发场景下，如果采用删除缓存的策略，很容易引发缓存击穿，此时应该选择最终一致性方案。

* 问题二：「读写分离 + 主从复制延迟」

  例如：

  ```
  线程t1写主库 price = 100。
  线程t1删缓存。
  线程t2读缓存为空。
  线程t2读从库，由于此时主库还没有同步数据到从库，存在延迟，因此线程t2读到旧数据 price = 50。
  线程t2将旧数据写缓存 price = 50。
  ```

  解决问题二，可以采用「**延迟双删**」策略。

  线程t1在「主库完成同步数据到从库之后」在删一遍缓存即可，这样后续线程在从库读到的都是最新的数据。

  因此这里延迟双删延迟的时间要大于「主从复制延迟」的时间。

<br />

**Cache Aside方案也存在问题为什么很多系统最终还是选择它？**

因为它在一致性、复杂度、性能之间，取得了非常好的平衡。它并不是绝对不会出错，但它足够简单，足够稳定，异常概率也足够低。因此「先更新 DB，再删除缓存」，最终成为了业界最主流的缓存一致性方案。



> 可以发现，在「更新缓存」的方案，我们讨论的都是写写并发的异常场景。在「删除缓存」的方案，我们讨论的都是读写并发的异常场景。原因是：
>
> 在「更新缓存」的方案，由于缓存是一直存在的，所以读线程只会读不会写缓存。由写线程负责更新缓存，因此写写并发存在的顺序问题会诱发缓存相互覆盖的问题。
>
> 在「删除缓存」的方案，写线程只会删除缓存，由于删除操作天然幂等，所以写写并发不会影响缓存的最终结果。更新缓存的动作由读线程在cache miss的时候触发，当读线程和写线程并发的时候，由于执行顺序问题，就会导致缓存中的值和DB中的值不一致。
>
> 所以，本质是：在「更新缓存」的方案中，写写并发是主要矛盾。在「删除缓存」的方案，读写并发是主要矛盾。



## 6. 「第二步失败」怎么办？

接下来我们继续讨论如何避免出现「只有第一步成功，第二步失败」的问题。

我们前面也说到了，失败大不了就重试嘛，重试也是有讲究的，比如说你是选择同步重试还是异步重试？

### 6.1 同步重试

同步重试就意味着你失败了立即重试，立即重试的问题在于：

* 立即重试很大概率还会失败
* 立即重试的次数设置为多少比较合理
* 重试会一直占用着这个线程资源，无法服务其它客户端请求
* 重试的过程中服务宕机了，这个重试请求就丢失了

所以同步重试的问题本质是**业务线程，不应该长期承担不确定操作**。

啥叫不确定操作？例如：

- 网络恢复什么时候成功？
- Redis 什么时候恢复？
- 重试多少次才够？

这些其实都是不可预测的事情，不应该由主业务线程长期承担。

因此同步重试，并不是高并发系统里优雅的解决方案。



一般我们都会采取异步重试，异步重试有两种方案：

* 采用MQ进行重试
* 订阅binlog+MQ来实现重试

### 6.2 采用MQ来进行重试

更新完DB之后，可以把删除缓存的请求封装为一条消息，写入MQ，让消费者异步地去删除缓存。

这样不会占用着主线程资源，而且就算服务宕机了，消息也不会丢失，可靠性很高。

![先改DB后删缓存 引入MQ异步删缓存](database-cache-consistency.assets/先改DB后删缓存-引入MQ异步删缓存.svg)

缺点：

* 如果应用程序恰好在写DB成功之后，发送MQ之前挂了，则这条消息就丢了，服务重启之后也只能等缓存自动过期。

  一种解决方案是先同步删一次缓存，只有当删除失败的时候再写入MQ，写缓存和写MQ同时失败是一个非常小概率的事件。但这依然也只是尽可能降低概率，无法保证消息一定能被写入MQ。

  另一种解决方案是通过引入本地消息表来保证消息一定被写入MQ，伪代码如下：

  ```
  事务内：
  1. update DB
  2. 写 message_table
  
  事务提交后：
  异步扫描 message_table 发 MQ
  ```

  但也提高了代码维护的复杂度。

* 对业务代码侵入性比较强，每个写DB操作后都要写一段发MQ的代码。

### 6.3 订阅binlog+MQ来实现重试

这种方案利用了 Canal 等工具，去监听数据库的Binlog，一旦发现有更新，就推送到 MQ。

Canal工具的底层原理：master节点主从同步的时候会将binlog同步到各个从节点，以便于从节点回放binlog同步数据。Canal会模拟成MySQL集群中的一个salve节点，从而实时获取binlog日志。

下图是订阅binlog+MQ来实现重试的完整流程

![订阅binlog发送到MQ](database-cache-consistency.assets/订阅binlog发送到MQ.svg)

优点：

* 对业务代码侵入性低。主业务代码只管操作数据库，完全不需要关心缓存的事儿。数据库写成功了，Binlog 必然产生，canal会自动投递到MQ，消费者自然可以获取到消息继而去操作缓存。
* 消息不会丢。只要数据库更新成功，binlog就一定存在。就算canal或者MQ挂了，重启之后依然可以根据消费者组的offset知道上一次消费到哪里，依然可以从上一次断开的地方继续往下执行。

缺点：

* 维护起来更加复杂，需要引入canal组建。



### 6.4 引入MQ重试需要考虑的问题

使用消息队列，需要考虑以下问题：

#### 1️⃣如何避免重复消费问题，保证消息只消费一次

MQ有三种投递语义：

* At most once: 消息在传递时，最多会被送达一次。换一个说法就是，没什么消息可靠性保证，允许丢消息。

  一般都是一些对消息可靠性要求不太高的监控场景使用，比如每分钟上报一次机房温度数据，可以接受数据少量丢失。

* At least once: 消息在传递时，至少会被送达一次。也就是说，不允许丢消息，但是允许有少量重复消息出现。

* Exactly once：消息在传递时，只会被送达一次，不允许丢失也不允许重复，这个是最高的等级。


现在常用的绝大部分消息队列提供的服务质量都是 `At least once` ，RocketMQ、RabbitMQ 和 Kafka 都是这样。也就是说，消息队列很难保证消息不重复。

所以我们无法从MQ层面去解决重复消费的问题，只能由消费者来保证幂等性。



例如，当前场景是我们要通过MQ异步更新缓存，如何保证一条消息只被消费一次？

解决方案：在Redis存储数据的时候也存储数据的版本（或者时间戳），投递消息的时候也记录消息的版本号。消费者从MQ获取到消息的时候，先检查一下消息的版本号，如果该版本号是小于等于当前Redis存储的数据的版本号，说明该消息已经被消费过，则不会再次消费。



#### 2️⃣顺序性

例如当前场景采用的是「先更新DB + 后通过MQ异步更新缓存」的策略

你先把商品的价格改为100，然后又改为200。如果消息无序的话，有可能最终的执行顺序是先改为200，后改为100，这样肯定是不能接受的。怎么解决？

* 方案1：通过消息队列保证顺序性。

  Kafka能够保证同一个partition中的消息是按顺序消费，因此可以把同一条数据的变更发送到同一个partition内部，则如果写入的顺序是先改为100后改为200，则消费者的消费顺序也一定是先改为100后改为200。

* 方案2：版本号/时间戳策略。

  如果你使用的MQ无法保证顺序性，使用版本号/时间戳策略可是可行的。

  例如：「消息1：把价格改为100 version=1」「消息2：把价格改为200 version=2」

  如果消费者先消费了消息2，把价格改为了200，后拿到消息1的时候发现version=1 < 当前数据的version=2 则认为这条消息已经过期了，不会再消费。



## 7. 可以保证强一致性吗

我们引入 Redis 缓存的初衷是什么？是为了极致的性能和高吞吐。

如果要求「强一致性」，唯一的方案是引入**分布式读写锁**。在写请求发生时，强行阻塞所有的读请求。

然而，锁会带来严重的线程挂起和串行化吞吐，这直接抹杀了 Redis 作为内存缓存的高并发优势，与我们引入缓存的初衷**背道而驰**。



**结论：** 既然选择了缓存带来的高性能，就必须容忍常态下存在短暂的不一致窗口。

我们的架构设计目标，不是消灭不一致，而是**如何通过各种补偿机制，将这个不一致的时间窗口压缩得越短越好，实现「最终一致性」**。
